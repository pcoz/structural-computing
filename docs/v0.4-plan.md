## 3. Deliverable 1 — MGI for general signatures

### 3.1 What ships

A populated `is_realisable` field on `HolographicBasisResult` for
**general (non-symmetric) signatures at arity 4**, computed via the
matchgate-identity check from `holant-tools.non_symmetric`. For
arity ≥ 5 the only currently-shipped identity is the augmented
weight-1 case; we ship that as a **necessary-but-not-sufficient**
check, document the limitation, and defer the full d-admissibility
framing to v0.5.

### 3.2 API delta

```python
# Today (v0.3)
result = hbp.transform_signature_general(values, arity=4)
result.is_realisable                # -> None (deferred)

# After v0.4
result = hbp.transform_signature_general(values, arity=4)
result.is_realisable                # -> True / False
result.realisability_check          # -> "matchgate_identity_arity_4"
                                    #    or "augmented_weight_1" (arity >= 5)
                                    #    or "deferred" (no check available)
```

New fields on `HolographicBasisResult`:

- `realisability_check: str` — names which identity was applied (so the
  caller knows whether the answer is *sufficient* or just *necessary*).
- (Optional, low priority) `subvariety: object` — the
  `realizability_subvariety_non_symmetric` result for callers who want
  the polynomial system itself, not just a boolean.

## 4. Deliverable 2 — Lipton-Tarjan auto-separator

### 4.1 What ships

A new `auto=True` mode on `PlanarSeparator` that discovers a
separator of size `O(√|V|)` automatically via the
Lipton-Tarjan 1979 planar separator theorem, eliminating the
requirement that the user supply `(separator, side_a, side_b)`.

### 4.2 API delta

```python
# Today (v0.3) — user-supplied
ps = PlanarSeparator(separator={2, 3}, side_a={0, 1}, side_b={4, 5})
plan = ps.decompose(graph)

# After v0.4 — auto-discovery
ps = PlanarSeparator(auto=True)
plan = ps.decompose(graph)        # internally calls Lipton-Tarjan on graph["rotation"]

# Orchestrator hint:
orch.evaluate(graph, "matching_count",
               hints={"planar_separator": "auto"})    # NEW: shorthand for auto=True
```

### 4.3 Lipton-Tarjan algorithm sketch

Standard reference: Lipton & Tarjan, *A Separator Theorem for Planar
Graphs*, SIAM J. Appl. Math. 36(2), 1979. For a planar graph G on n
vertices, there is a separator S of size `≤ 2√(2n)` whose removal
leaves two components each of size `≤ 2n/3`. The algorithm is `O(n)`:

1. BFS-layer the graph from an arbitrary root → level sets `L_0, L_1, ...`.
2. Find a level `L_t` of size `≤ √n` such that the levels above it
   contain `≤ 2n/3` vertices and the levels below contain `≤ 2n/3`
   vertices. `L_t` is the separator.
3. If no single level works, use a more careful argument involving a
   pair of levels and a spanning-tree fundamental cycle.

The algorithm needs the **rotation system** (which the graph dict
already carries after `NormaliseGraphFormat`) to compute the levels
correctly on a planar embedding.

### 4.4 Concrete changes

| File | Change |
|---|---|
| `structural_computing/decompose.py` | New private function `_lipton_tarjan_separator(rotation) -> (separator, side_a, side_b)`. `PlanarSeparator.__init__` accepts `auto: bool = False`. `decompose()`: if `auto`, call the new function on `problem["rotation"]` before validation. |
| `structural_computing/orchestrator.py` | Phase 4.8: accept `hints["planar_separator"] = "auto"` as shorthand for the auto path. Existing dict form `{"separator": ..., "side_a": ..., "side_b": ...}` still works. |
| `tests/test_decompose.py` | (a) Lipton-Tarjan on a planar grid (k×k) — verify the returned separator is valid (covers an edge cut) and the resulting matching count matches user-supplied. (b) Round-trip: auto-separator → matching count = brute force on K₄ tetrahedron, planar bipartite, planar grids. (c) Size bound: separator size ≤ `2√(2|V|)` on the test graphs (Lipton-Tarjan theorem invariant). |
| `tests/test_orchestrator.py` | Add a test exercising the `"auto"` hint shorthand. |
| `CHANGELOG.md` + `docs/architecture.md` | Move from roadmap to v0.4 entry. Update §7.3 (PlanarSeparator status). |

### 4.5 Acceptance criteria

- `_lipton_tarjan_separator` returns a valid partition: union covers V,
  pairwise disjoint, side_a and side_b sub-graphs have no direct edge
  between them.
- Separator size is bounded by `2√(2|V|)` on every test graph (the
  theorem's invariant).
- Matching count via auto-PlanarSeparator equals matching count via
  brute force on at least 5 distinct planar graphs (K₄, planar
  bipartite, 3×3 grid, 4×4 grid, planar weighted).
- No regression on existing user-supplied PlanarSeparator tests.

### 4.6 Estimated effort

**3–5 days.** Implementing Lipton-Tarjan correctly from scratch is real
work: the layering pass is straightforward, but the
spanning-tree-fundamental-cycle backup case has corner cases (degenerate
trees, multi-component graphs). The right approach is to ship the
"good case" (single-level separator suffices) first with a clear
honest-stop fallback to user-supplied for the corner cases, then add
the spanning-tree backup as a polishing pass before release.

## 5. Deliverable 3 — SRP search polish

### 5.1 What ships

A documented corpus of matchgate-realisable symmetric signatures that
the current `discover_basis` *fails to find a basis for* despite the
existence of one (i.e., the order-2 recurrence is satisfied but the
canonical-bases-plus-grid path doesn't converge), plus an upgrade to
the search strategy that handles them.

### 5.2 Why this matters

`discover_basis` today returns `None` in two distinct cases:

1. **No basis exists.** The signature doesn't satisfy the order-2
   recurrence — Cai-Lu 2.5 says no basis can rescue it. The
   realisability gate at the top of `discover_basis` catches this.
   This is correct behaviour.
2. **A basis exists but the search misses it.** The signature
   *does* satisfy the recurrence, but the canonical candidates and the
   `[-2, +2]` 3D grid don't land near it. Today the function returns
   `None` indistinguishably from case 1. **This is the case v0.4
   needs to fix.**

The fix has two parts: (a) a way to *recognise* the user has hit
case 2 vs case 1, and (b) a search upgrade that resolves case 2.

### 5.3 Approach

#### 5.3.1 Diagnostic: distinguish case 1 from case 2

Use the order-2 recurrence kernel directly. When `discover_basis` exits
without success, check the kernel coefficients `(a, b, c)` returned by
the realisability gate. The kernel encodes the matchgate-standard form
the signature *should* land in; from `(a, b, c)` we can analytically
derive the basis that puts it in standard form via a closed-form
(diagonalisation of the recurrence matrix). This bypasses the search
entirely for the well-conditioned cases.

#### 5.3.2 Build the failure corpus

A documented set of test signatures, each:

- satisfies the order-2 recurrence (realisable on *some* basis), and
- is NOT found by the current `discover_basis`.

Source candidates: every symmetric signature of arity 3–5 whose
recurrence kernel falls outside the `[-2, +2]^3` grid's coarse-step
neighbourhood. Generate by sampling kernel coefficients on a wider
range, building the corresponding signature, and checking the current
search returns `None`.

#### 5.3.3 Upgrade the search

Once we have the corpus, two complementary strategies:

(a) **Analytic shortcut.** Derive `T` directly from the recurrence
kernel via the standard "diagonalise the companion matrix" construction.
For most cases this gives the answer in O(1) without any search.

(b) **Extended canonical list + adaptive grid.** Add more known
matchgate-unlocking bases (e.g., the discrete-Fourier basis for cyclic
signatures, the parity-conjugate basis); when the grid runs and the
best candidate is on the grid boundary, expand the grid by a factor of
2 and retry.

### 5.4 Concrete changes

| File | Change |
|---|---|
| `structural_computing/compose.py` | New private method `_basis_from_recurrence_kernel(a, b, c)` returning a 2x2 T (closed form). `discover_basis` calls this *before* the canonical-bases sweep; if its output puts the signature in standard form, return immediately. The existing search remains as a fallback. |
| `tests/test_compose.py` | Add corpus tests: ~10 signatures from the failure corpus, each must find a basis. |
| `tests/test_compose.py` | Stress test: generate 100 random recurrence kernels in `[-10, +10]^3`, build the corresponding signature, verify `discover_basis` finds a basis for each. |
| `CHANGELOG.md` | Note: SRP search is now closed-form for symmetric signatures when the recurrence kernel is well-conditioned; grid search remains as the fallback. |

### 5.5 Acceptance criteria

- Every signature in the failure corpus has a basis discovered.
- 100/100 random recurrence-kernel-derived signatures are handled.
- No regression: the existing 13 `discover_basis` tests still pass.
- The analytic shortcut returns a `T` with `det(T) != 0` and
  `T` applied to the signature has matchgate-standard distance
  `< 1e-6` on the corpus.

### 5.6 Estimated effort

**2–4 days.** The closed-form derivation is straightforward
linear algebra. The corpus-building and stress test take the bulk of
the time.

## 7. Out of scope for v0.4 (deferred to v0.5+)

Explicit non-goals — these are real work, but not in this release:

- **Full Cai-Lu §4 d-admissibility** for `d ≥ 1` on general signatures.
  The current MGI check covers arity 4 fully (even + odd parity
  identities) and arity ≥ 5 partially (augmented weight-1 only).
  The full d-admissibility framing — where matchgate identities cut
  subvariety dimension down to 0/1 — is documented in
  `MATCHGATE_IDENTITIES.md` "Deferred" section and is a v0.5 target.
- **Hardness recognition utility.** A by-product utility that flags
  signatures lying outside the matchgate-realisable locus on every
  parity branch and chart. Useful, not blocking v0.4.
- **The "viewing-frame inversion" research direction.** Per the
  2026-05-24 decision log in `admissibility-geometry`, this is the
  *research-track* v0.4 ambition (enlarge the description apparatus
  for problems relative to `M`, not the realisability theory itself).
  Lives in `tools/admissibility/matchgate_rank.py` in the research
  repo. Different but compatible — feeds future *packaged* releases.
- **`structural-computing-bench v0.4`** (README + matplotlib plots).
  Separate repo, separate release track. Blocks on PyPI publication
  of this repo (see §6).
- **Wrapper-method consolidation.** `StructuralComputer` today
  duplicates some leaf-evaluator logic that the orchestrator already
  has. Refactoring `StructuralComputer` to delegate everything to
  `Orchestrator.evaluate(...)` is good housekeeping but not in v0.4
  scope — risk of breaking the wrapper API for users on v0.3.

Goals 1, 2, and 3 are **independent** and can run in parallel:

```
MGI  ─────[2-3 days]──────┐
                          ├──► Tests pass ─► CHANGELOG + docs ─► tag ─► PyPI
Lipton-Tarjan ──[3-5 d]───┤                                              │
                          │                                              │
SRP polish ──[2-4 days]───┘                                              │
                                                                         │
PyPI v0.3 dry run ────[0.5 d, in parallel any time]───►──────────────────┤
                                                                         │
                                                                         ▼
                                                                  v0.4.0a1 RELEASED
```

Total elapsed time depends on parallelisation. Sequentially:
~2 weeks. With two work streams: ~1.5 weeks. Single-stream agent
implementing all four: ~10 working days.

**Suggested order for a single-stream implementation**:

1. **MGI first** (2-3 days) — engine is ready, fastest unlock,
   highest user-facing impact.
2. **PyPI v0.3 dry run in parallel** with MGI (half day, can be done
   in any quiet window).
3. **SRP polish** (2-4 days) — moderate effort, independent of MGI.
4. **Lipton-Tarjan last** (3-5 days) — most algorithmic work, but
   doesn't gate anything else.
5. **Release** — bump version to `0.4.0a1`, update CHANGELOG, tag,
   publish.
