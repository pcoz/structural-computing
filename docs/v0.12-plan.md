## 1. Headline goal

Eliminate the duplicate evaluation logic that had built up between
`StructuralComputer` (the friendly one-liner wrapper) and
`Orchestrator` (the full engine).

Before v0.12:
- `StructuralComputer.count_matchings(graph)` did its own
  normalisation, classification, and dispatch (calling
  Kasteleyn-Pfaffian or brute force directly).
- `Orchestrator.evaluate(problem, "matching_count")` did
  approximately the same thing through `_matching_count_leaf`.
- Same pattern for ~15 other methods.

After v0.12:
- `StructuralComputer.count_matchings(graph)` builds a problem
  dict and calls `self._delegate(problem, "matching_count")`.
- `Orchestrator.evaluate(problem, "matching_count")` is the
  single source of truth.

The wrapper becomes thin ergonomic shims around the orchestrator;
any new question registered in `DEFAULT_LEAF_REGISTRY` is
automatically reachable through a small wrapper method.

## 2. Deliverable detail

### Deliverable 1 — Delegation infrastructure

`StructuralComputer.__init__` now creates a single `Orchestrator`
instance:

```python
def __init__(self):
    self._last_classification = None
    self._orchestrator = Orchestrator()  # NEW in v0.12

def _delegate(self, problem, question):
    """Run a single-leaf question through the orchestrator;
    return the bare answer; sync _last_classification."""
    result = self._orchestrator.evaluate(problem, question)
    self._last_classification = result.classification
    return result.answer
```

### Deliverable 2 — Matching + tropical methods

Refactored to delegate:

- `count_matchings` — now goes via `(T2/T4, matching_count)`.
  The leaf was upgraded to use Kasteleyn-Pfaffian (FKT) when a
  rotation system is present + the graph is planar; brute force
  otherwise. This preserves the wrapper's pre-v0.12 smart path
  while making the leaf the single source of the smart-path
  logic.
- `tail_probability` — now via `(T2/T4, tail_probability)`.
  `p_fail` rides inside the problem dict.
- `witness` — now via `(T2/T4, witness)`.
- `single_points_of_failure` — now via
  `(T2/T4, single_points_of_failure)`.
- `min_weight_matching` (added v0.10) — now via
  `(T2/T4, min_weight_matching)`.
- `min_cost_schedule` (v0.10), `min_cost_flow` /
  `min_cost_roster` / `min_cost_dedup` /
  `tropical_instance_coordinates` (v0.11) — all delegate to
  their respective leaf evaluators.

### Deliverable 3 — Constraint + signature methods

Refactored to delegate (with pre-coercion of A/b/Q/c to numpy
arrays since the classifier needs numpy):

- `count_solutions` — now via `(T0/T1, count_solutions)`.
- `find_witness_solution` — now via `(T0/T1, find_witness)`.
- `list_solutions` — now via `(T0/T1, list_solutions)`.

The signature methods (`classify_function`, `matchgate_rank`,
`is_matchgate_realisable`) were already minimal pass-throughs to
`classify_signature`; they were not refactored since they don't
go through leaf evaluators.

### Orchestrator-side classifier extension

`Orchestrator._classify_problem` was extended to recognise two
additional problem-dict shapes that the wrapper produces:

1. **Tropical instance-based dicts** (`{"instance": ...}`) —
   routed to T2 with `problem_kind = "tropical_instance"`. The
   tropical family of leaf evaluators is registered under T2 and
   T4; T2 routing is sufficient since the leaves themselves
   inspect the instance type.
2. **Graph dicts without rotation** (`{"vertices": ...,
   "edges": ...}`) — routed to T2 with `problem_kind =
   "graph_no_rotation"`. The leaf evaluator (e.g.,
   `_matching_count_leaf`) treats this as a generic graph and
   falls back to brute force if no embedding is available.

## 3. Methods NOT refactored (intentionally)

Some methods are not single-leaf evaluations and shouldn't be
delegated:

- `classify`, `explain` — structural inspection.
- `compare` — meta-method that calls `tail_probability` twice.
- `audit` — meta-method that calls several wrapper methods.
- `count_matchings_hybrid` — multi-step reduction
  (HybridDecomposition).
- `classify_constraints`, `classify_function`, `matchgate_rank`,
  `is_matchgate_realisable` — direct classification calls.

Documented explicitly in the CHANGELOG.
