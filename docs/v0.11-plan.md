## 1. Headline goal

Close the tropical-wiring gap left open by v0.10. v0.10 wired the
two highest-impact tropical questions (`min_weight_matching` and
`min_cost_schedule`); v0.11 finishes the job for the remaining
domain-specific tropical primitives in holant-tools' surface.

After v0.11, every domain-specific tropical primitive in
`holant_tools` is reachable through the structural-computing
orchestrator AND the `StructuralComputer` one-liner wrapper.

## 2. Deliverables in detail

### Deliverable 1 ‚Äî Leaf evaluators for remaining tropical primitives

Four new leaf evaluators in `orchestrator.py`:

- **`_min_cost_flow_leaf`** ‚Äî exact polynomial-time minimum-cost
  flow on a `holant_tools.MinCostFlowInstance` (sources, sinks,
  capacity-constrained edges). Returns
  `{"cost", "flow", "feasible"}`.
- **`_min_cost_roster_leaf`** ‚Äî exact rostering: assign employees
  to shifts respecting headcount, certifications, and
  unavailability constraints, minimising total preference cost
  via the supplied `preference_fn`. Returns
  `{"cost", "roster", "feasible"}`.
- **`_min_cost_dedup_leaf`** ‚Äî exact record-to-entity-candidate
  assignment for entity deduplication (master-data management),
  minimising total dissimilarity via the supplied `similarity_fn`.
  Returns `{"cost", "assignment", "entity_groups", "feasible"}`.
- **`_tropical_instance_coordinates_leaf`** ‚Äî the one-call
  diagnostic: "is this `SchedulingInstance` structurally well-
  suited for tropical optimisation?" Returns the
  `TropicalInstanceCoordinates` dataclass directly (the fields
  ARE the diagnostic output).

Registry entries added under (T2, ...) and (T4, ...) for each.
The instances themselves are domain-specific dataclasses; the
classifier dispatches based on the question, not the instance
type.

### Deliverable 2 ‚Äî StructuralComputer wrapper methods

Four new methods on `StructuralComputer`:

- `sc.min_cost_flow(instance) -> dict`
- `sc.min_cost_roster(instance, preference_fn) -> dict`
- `sc.min_cost_dedup(instance, similarity_fn) -> dict`
- `sc.tropical_instance_coordinates(instance, cost_fn, *,
    compute_field_distance=False) -> TropicalInstanceCoordinates`

Four new pytest tests in `tests/test_smoke.py` covering: flow
golden path, roster pairing (Alice/Bob preferences), dedup
pairing, coordinates diagnostic on uniform-cost instance.

## 3. What's still carried forward (NOT v0.11 scope)

The lower-level polymorphic Pfaffian + multilinear evaluators
(`pfaffian(M, semiring=...)`,
`holant_sum_of_pfaffians(terms, semiring=...)`,
`holant_genus_g(matrices, genus, semiring=...)`,
`holant_planar(M, semiring=...)`) remain accessible via direct
import from `holant_tools`. They don't need orchestrator wiring
because they're matrix/tensor-level operations rather than
problem-level questions:

```python
import holant_tools
holant_tools.pfaffian(M, semiring=holant_tools.TropicalMinPlusSemiring)
```

After v0.11, the tropical-Holant capability is fully reachable
through the orchestrator + wrapper for every domain-specific
primitive.
