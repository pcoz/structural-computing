## 1. Headline goals

Three flavours of polish around the v1.0.0 milestone:

1. **Production calibration data** for the entire current
   leaf-evaluator surface, not just the v0.1–v0.3-era subset.
2. **Discoverable documentation** via the Diátaxis structure —
   so downstream adopters can find what they need by intent
   (learn / do / look up / understand).
3. **Stability-tier promotions** for symbols that have hardened
   across the v0.4–v0.13 development cycle.

## 2. Deliverable detail

### Deliverable 1 — Extended bench + calibration runs

**Engine side** (`structural-computing-bench/bench/`):

- `bench/problems.py`: six new problem generators
  (`make_weighted_bipartite_graph`, `make_scheduling_instance`,
  `make_flow_instance`, `make_rostering_instance`,
  `make_dedup_instance`, `make_cpsat_cardinality_model`). Each
  returns a problem dict in the shape its corresponding leaf
  evaluator expects.
- `bench/calibrate.py`: imports the seven new leaves from
  `structural_computing.orchestrator`; adds seven new
  `DEFAULT_SIZES` entries; adds seven new `_add(...)` calls in
  `calibrate_default_evaluators`.

**Calibration run.** Ran on the current machine
(`AMD64-Windows-11`, Python 3.13.13) with `--repeat 3`. All 11
leaves fitted; data file written to
`calibration_data/default.py` (replaces the v1.0.0 4-leaf
version).

**Fit summary** (curves below 0.5 RMS, no outliers):

| Leaf | Model | Slope (b) |
|---|---|---|
| count_solutions (T0) | exponential | +0.10 |
| matchgate_rank (T2) | power_law | +3.44 |
| matching_count (T2) | power_law | +1.00 |
| matching_count (T4) | power_law | −0.32 |
| min_cost_dedup (T2) | exponential | +0.13 |
| min_cost_flow (T2) | exponential | +0.01 |
| min_cost_roster (T2) | exponential | +0.16 |
| min_cost_schedule (T2) | power_law | +0.06 |
| min_weight_matching (T2) | power_law | +2.39 |
| rewrite_cpsat_model (T2) | exponential | +0.09 |
| tropical_instance_coordinates (T2) | power_law | +1.37 |

The min_weight_matching ~n^2.4 fit is consistent with the
expected Hungarian `O(n^3)` (the calibration sweep size range is
small so the fitted exponent under-estimates). Production runs
can use the data file directly via `apply_calibration(...)`.

### Deliverable 2 — Diátaxis docs restructure

`docs/` directory now organised by Diátaxis quadrant:

```
docs/
├── README.md                  — top-level index
├── STABILITY.md               — per-symbol stability contract
├── architecture.md            — deep-dive system reference (existing)
├── tutorial/
│   └── getting-started.md     — 30-minute end-to-end walkthrough
├── how-to/
│   ├── min-cost-scheduling.md
│   └── cpsat-preflight.md
├── reference/
│   └── api.md                 — every public method on StructuralComputer
├── explanation/
│   └── tropical.md            — semiring-choice argument
└── v*.* plan docs              — historical release plans
```

**Tutorial.** Six steps in 30 minutes: count → tail probability →
compare → min-weight matching → CP-SAT pre-flight → honest stops.
Every step has a runnable code block.

**How-to entries.** Each is a focused recipe for a specific
task — context-free (you know what you want, you need the
exact code). Min-cost scheduling covers the constraints
(`allowed_machines`, `time_windows`, `forbidden_edges`); CP-SAT
pre-flight covers `helped: bool` branching and verification.

**Reference page.** All 30+ public `StructuralComputer` methods,
grouped by capability era (v0.1–v0.3 / constraint / signature /
tropical / CP-SAT / hybrid / inspection), with one-line
descriptions + cross-links to STABILITY.md.

**Explanation page.** "Why tropical works" — the semiring-choice
argument. Useful to readers who want to understand WHY the
framework is structured this way before committing to it.

**Main README** updated to point into the new structure.

### Deliverable 4 — Experimental-tier hardening

Five Experimental-tier symbol groups promoted to Stable as of
v1.1.0:

1. **Reduction / composition / decomposition base interfaces**:
   `Reduction`, `ReductionResult`, `ReductionPlan`,
   `ReductionNotApplicable`, `Composition`, `CompositionPlan`,
   `Decomposition`, `DecompositionPlan`.
2. **Pipeline-router framework**: `Stage`, `Route`,
   `StageRecord`, `Trace`, `run_pipeline`, `run_pipeline_streaming`.
3. **Verifier helpers**: `brute_force_count_matchings`,
   `brute_force_weighted_matching_sum`, `satisfies_gf2_affine`,
   `enumerate_satisfying_assignments`, `gibbs_expectation_brute`,
   `verify_pipeline`.
4. **Trace aggregator**: `RichTrace`, `RegimeChange`.
5. **Replay cache**: `ReplayCache`, `cached_runner`,
   `default_key`.

The **specific implementation classes** stayed Experimental:
`NormaliseGraphFormat`, `CrossingElimination`, `HybridDecomposition`,
`RationaliseWeights`, `LinearCombination`, `Projection`,
`HolographicBasisPair`, `BranchSum`, `ShannonExpansion`,
`TreewidthBoundedDP`, `PlanarSeparator`, `RecursiveCircuitCut`.

**Asymmetry documented in STABILITY.md.** Subclassing the base
interfaces gets a Stable contract; depending on the EXACT
internals of a specific implementation does not.
