## 1. Headline goals

The framework has been GitHub-tracked for v0.6 but the only package
live on PyPI is `holant-tools v0.5.0` (released 2026-05-27, before
v0.6). v0.7's single arc:

1. Land **holant-tools v0.6.1** on PyPI (the v0.6 augmented-Plücker
   work — both m=1 and m=3 configurations).
2. Land **structural-computing v0.6.0a1** on PyPI for the first time
   (this is *publication* — the package has never been live).
3. Land **structural-computing-bench v0.1.0a1** on PyPI for the first
   time (it was blocked on structural-computing's upload, since the
   bench depends on `structural-computing` runtime).

After v0.7, downstream consumers can:

```
pip install structural-computing
pip install structural-computing-bench
```

and the holant-tools dependency resolves transparently.

## 2. Why now

Per the v0.6 decision log: "PyPI publication (still pending external
action; dist artefacts for v0.5.0a1 built but not uploaded; v0.6.0a1
dist will need fresh build)." v0.7 directly addresses that follow-on.

Additionally, the v0.6 PyPI gap is *blocking* `structural-computing-
bench` from being usable as a `pip install`. Until the framework is
live, the bench is a GitHub-only research artefact.

## 3. Deliverables in detail

### Deliverable 1 — Rebuild + verify holant-tools v0.6.1 dist

Stale `dist/holant_tools-0.5.0*` artefacts removed. Fresh `python
-m build` produced `holant_tools-0.6.1.tar.gz` + `holant_tools-0.6.1

clean venv: `import holant_tools` works; `__version__ == '0.6.1'`;
`matchgate_identities_arity_n_odd_augmented` imports correctly.

### Deliverable 2 — Trusted Publisher workflow

`.github/workflows/publish.yml` added to holant-tools, mirroring
structural-computing's existing one. Uses `pypa/gh-action-pypi-publish
@release/v1` with `permissions: id-token: write`. Triggers on GitHub
Release publication OR via `workflow_dispatch` for TestPyPI smoke
checks.

**User-action prerequisite for D3:** configure the holant-tools
Trusted Publisher entry on PyPI (Account Settings → Publishing). The
workflow filename, repo, and environment must match exactly.

### Deliverable 3 — Upload holant-tools v0.6.1 to PyPI [USER-ACTION]

After D2's workflow is live and the Trusted Publisher is configured:

```
gh release create v0.6.1 -t "v0.6.1 — augmented Plücker m=3 extension" \
  --notes "See CHANGELOG.md"
```

Or use workflow_dispatch with target=pypi.

**Verification.** From a clean venv:

```
pip install holant-tools==0.6.1
python -c "import holant_tools; print(holant_tools.__version__)"
# Expected: 0.6.1
```

### Deliverable 4 — Rebuild + verify structural-computing v0.6.0a1 dist

Stale `dist/structural_computing-0.5.0a1*` artefacts removed. Fresh
`python -m build`. `twine check dist/*` PASSED. Wheel METADATA

### Deliverable 5 — Upload structural-computing v0.6.0a1 to PyPI [USER-ACTION, depends on D3]

structural-computing has had `.github/workflows/publish.yml` since
v0.5; the Trusted Publisher entry on PyPI was configured during the
v0.5 dist-prep work.

After D3 completes (holant-tools v0.6.1 live on PyPI):

```
gh release create v0.6.0a1 -t "v0.6.0a1 — first PyPI release" \
  --notes "See CHANGELOG.md"
```

**Why the order matters.** structural-computing depends on
`holant-tools>=0.6.1`. If structural-computing uploads first, anyone
running `pip install structural-computing` from a clean environment
will fail with "no matching distribution for holant-tools>=0.6.1".

**Verification.** From a clean venv:

```
pip install structural-computing
python -c "from structural_computing import StructuralComputer"
```
